# Автоматизация обновлений

## Базовый поток

1. В продуктовом репозитории завершается merge в основную ветку.
2. Workflow вычисляет список изменённых файлов.
3. Список файлов передаётся в wiki-репозиторий через `repository_dispatch`.
4. Wiki определяет затронутые функции через `docs-map/features.json`.
5. Для этих функций обновляются статьи и планируются новые скриншоты.
6. Создаётся pull request на проверку.

## Почему через PR, а не напрямую в production

- изменение кода не всегда означает изменение пользовательского поведения
- тексты и скриншоты должны проходить просмотр
- иногда одно техническое изменение затрагивает несколько сценариев

## Что автоматизировать в первую очередь

- определение затронутых функций
- пересъёмку экранов
- создание черновиков статей
- уведомление владельцев функции
